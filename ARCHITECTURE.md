# High-Level Architecture

## Overview
This portfolio uses a unique approach where page content is generated by executing BrainFuck code. The architecture follows a clean separation between code interpretation, HTML generation, and styling.

## Components

### 1. **index.html**
- Entry point for the application
- Provides HTML structure (DOCTYPE, head, body)
- Contains `<div id="app">` for dynamic BF-content injection
- **Music Widget**: Persistent `<div class="music-player">` overlaid above the app layer
- Loads `interpreter.js` for BF runtime

### 2. **interpreter.js**
The core runtime engine with four main functions:

#### `runBrainfuck(code)`
- Executes BrainFuck bytecode
- Maintains a 30,000-byte tape (memory)
- Returns HTML string output
- Handles all BF operations: `> < + - . [ ]`

#### `loadBF(path)`
- Async function that fetches `.bf` files
- Returns file contents or 404 error HTML
- Handles errors gracefully

#### `loadPage(path)`
- Loads BF file based on route
- Executes BF interpreter on file content
- Caches results for performance
- Injects HTML into `<div id="app">`

#### Navigation Functions
- `navigate(path)`: Updates URL and loads page
- `popstate` listener: Handles browser back/forward
- `click` listener: Intercepts navigation links

### 3. **BrainFuck Files (.bf)**
Located in `/bf/` directory:
- Each `.bf` file is pure BrainFuck source code
- When executed, produces HTML output
- Examples:
  - `/bf/home.bf` → Home page HTML
  - `/bf/about.bf` → About page HTML

### 4. **CSS Styling**
- External stylesheet or inline `<style>` tag
- Styles the HTML generated by BrainFuck
- Controls layout, positioning, colors, typography
- Treated independently from page generation logic

### 5. **Music Widget (Persistent Layer)**
Located outside the BF architecture at the DOM level:
- **Persistent**: Survives across page changes (not re-rendered by BF)
- **Fixed Position**: Bottom-left corner with slide-in/out animation
- **z-index: 9999**: Sits above all BF-generated content
- **Features**:
  - Play/Pause controls
  - Previous/Next track navigation
  - Volume slider with dynamic icon
  - Album art emoji display
  - Track name and artist info
  - Smooth slide animation (0.35s cubic-bezier)
  - Professional gradient background (purple theme)
  - Toggle button to minimize/restore
  - Responsive design (full width on mobile)

Music widget lifecycle:
1. Widget exists in DOM before BF page renders
2. User interacts with controls (independent of page content)
3. Page navigation doesn't affect widget state
4. Widget persists across all rendered pages

## Data Flow

```
Routing
  ↓
routes[path] resolve
  ↓
Fetch .bf file → loadBF()
  ↓
Execute BrainFuck → runBrainfuck()
  ↓
HTML string created
  ↓
Cache result
  ↓
DOM update: app.innerHTML = html
  ↓
CSS renders final page
```

## Key Design Decisions

1. **Pure BrainFuck Generation**: Page content is 100% generated by BF execution
2. **Caching**: Results cached to avoid re-interpreting BF on repeat visits
3. **SPA Architecture**: Single-page app with client-side routing
4. **Separation of Concerns**: 
   - BF codes generates content (HTML)
   - CSS handles presentation (styling)
   - JavaScript manages routing and lifecycle
5. **Stateless Routes**: Routes map to `.bf` files; no server-side state

## Performance Considerations

- **Caching**: First load interprets BF; subsequent loads use cached HTML
- **Tape Size**: 30KB memory tape per BF execution
- **Loop Handling**: Proper bracket matching prevents infinite loops in most cases
